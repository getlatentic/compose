## What's left, in priority order

### #6 — Markdown coverage gaps in the renderer

The parser already understands tables, lists, blockquotes, HRs, task
lists, footnotes, YAML frontmatter, images. The **renderer** currently
treats them as plain body text (one segment with literal source). To
render them properly:

1. **Add new `BlockRule` trait** in `rules.rs` (mirror of `InlineRule`) — block-level dispatch from `BlockKind` to a renderer-specific projection.
2. **Built-in block rules**: `ListRule`, `BlockQuoteRule`, `HorizontalRuleRule`, `TableRule`, `TaskListItemRule`, `CalloutRule` (Obsidian `> [!NOTE]`).
3. **Extend `CanvasTextStyle` and `LineKind`** — `kind: "list-item" | "block-quote" | "horizontal-rule" | "table-row" | "callout"`. Bump the wire version.
4. **TS renderer**: register matching style specs; paint list bullets / quote bars / HR lines / table cell backgrounds.
5. **Tests**: extend `presentationPlanShape.test.ts` snapshot fixtures with one per construct; expand `spec_conformance.rs`'s canonical-construct list.

The registry (`PresentationRuleRegistry` in `rules.rs`, `SegmentStyleRegistry` in TS) is set up so each new rule is **register + implement** — no central match to grow. That's deliberately the design slice #4 set up.

**Estimated effort**: ~4-6 hours of careful work. Most of it is rendering decisions (do we show table grid lines? what does a callout border look like?) more than mechanics.

### #2 — Bob auth round-trip proof

Pieces already shipped (from prior session and earlier in this one):
- Settings → Test Bob action (desktop-only) verifies CLI install + saved key + runs a smoke command
- IPC boundary in `settingsClient.ts:63`
- Backend `settings.rs:96`
- Composer sized correctly (37px / 36px) in `global.scss:1123`

What's **not** done:
- An authenticated round trip with a real saved key — currently the keyring has no Bob key, so the smoke test reports "auth required" or 401.
- Auto-verify after save (Settings should call Test Bob automatically when user saves a key).
- Persistent verification badge so users see auth status without opening Settings.
- Startup verification when a key is configured.

**To close**:
1. Add a real Bob API key via Settings → save.
2. Run `pnpm tauri build` and exercise Settings → Test Bob — confirm `authenticated: true`.
3. Optionally wire auto-verify-after-save + startup check (UX polish).

**Blocker**: requires a real Anthropic API key (or whatever Bob accepts) — can't be done without user credential.

### Nice-to-haves spotted along the way

- **LineGeometry `top_of` could use `u32` math more aggressively** — the prefix-sum table is `Vec<u32>` but we cast to `usize` for the wire format. Tiny.
- **WASM bundle** — 532 KB / gzip 217 KB. Could shave ~30 KB by stripping `serde_json` if we switched the wire format to a hand-written serializer. Not worth it unless we hit a size budget.
- **The `_default` thread-local pattern** in `rules.rs` could be promoted to a generic "engine state" registry if more such caches accumulate. Defer until a third use case exists.

## Where to look first when picking up

1. **`docs/benchmarks/baseline.md`** — current perf state. Anything you change in the editor input / renderer / WASM should be re-run through `pnpm bench:baseline` and diffed.
2. **`docs/vellum-local-first-production-spec-v3.2.md`** — the architecture spec. Sections §9.4 (rule registry), §10.2/10.5 (semantic map), §6.4 (transactions) all map directly to shipped code.

4. **`src/features/editor/CanvasMarkdownEditor.tsx`** — the integration point between React state and the WASM handle. Most "the editor does X weirdly" bugs route through here.
5. **`crates/vellum_engine/tests/spec_conformance.rs`** — if you change the parser or AST, this is the safety net.
